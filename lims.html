Lab Management System

At Forage Genetics I was tasked with developing a system to increase the throughput of the lab processes. With the help of a team, we created a Java based lab information management system (LIMS) that tracks each step in the lab qPCR process by scanning the barcodes on the lab equipement. It then generates the necessary input files for a Quant Studio PCR machine. Finally the output of this machine is automatically analyzed and dosage calls are made using my Quadrophenia machine learning dosage calling algorithm. Each step is detailed in the following.

Request Form

A tool for requesting a new job in the lab

Our lab serviced many breeding programs throughout the country. In order to expedite this process, we created a web-based user interface. The first part of this process is called the Request Form. With this web form, users can request to be provided with DNA plates that they can place samples of tissue in. Moreover, this form tracks important information regarding the origin of this tissue so that the lab can adjust their processes and analysis. An example of this form is shown here.

Plate Setup File

A map for adding tissue samples to the DNA plates

Once a job is requested, the lab sends the DNA to the requestor for tissue sampling. This requires precisely placing the correct tissue samples into each of the 96 wells of each plate. Moreover, a given job could upwards of 40 plates. Thus, the LIMS generates a map for each plate so the tissue samplers know where to put each tissue. This plate also works as the first hard record in the information recording process. An example of this plate setup file is here. Note, proprietery information was removed for discretion.

API

The interface between the barcode scanner and LIMS

The primary source of the data management in this system is through barcode scanning. By scanning each piece of lab equipment during the process we can both track how complete the process is and ensure that the process is following the correct steps. Since barcodes are so prevelant in this system, we have also created special barcodes to act as the functions for each step. Thus the entire interface operates with barcode scans.

Quality Control

One of the functions that the LIMS can handle is keeping track of quality control notes

One of the main requests of the system is to properly handly quality control data tracking. Prior methods consisted of writing the quality of each DNA plate on the plate setup sheet and then filing that away never to be seen again. With the LIMS, we allow for tracking quality immediately upon scanning the DNA plate for processing. The image to here shows the screen for entering the quality data for the DNA plates.

Once this screen has been activated, scanning the DNA plate will shows the quality for all associated plates of the same job. This way the user can enter data for each plate since often all plates are observed simultaneously to detect variations. Moreover, they can change the quality by selecting that plate in the list box. Once they have entered the quality for each plate and indicated that they are finished, the quality control for each plate is saved to the server as a .csv file. Thus, the quality information is available throughout the lab process pipeline.

Assign to a PCR Plate

Sets of 4 DNA plates are assigned to a PCR plate and interlaced in a pre-defined order

Our qPCR machine utilized 384 well PCR plates. This means that 4 DNA plates had to be combined to make one PCR plate. Moreover, this process needed to be done in such a way as to minimize local batch effects (i.e. not simply placing the plates in each quarter). Thus, the LIMS allows for assigning 4 DNA plates to a PCR plate, then the remaining steps can track how these 4 plates were interlaced onto the PCR plate.

Once 4 DNA plates are assigned to a PCR plate, the results assignments are saved to a .csv file on the server. Moreover, we have multiple machines and the machine that was used can be assigned as well. Tracking the machine is an important step in quality control and measuring batch effects. Finally, updating this .csv file triggers an R script to generate input files for the PCR machines.

Generate a Batch Experiment

Once the plates have been assigned, input files can be generated for the qPCR machine

Once the quality has been tracked and the DNA plates assigned to a PCR plate, the PCR reaction can begin. We used an Applied Biosystems qPCR machine that runs with Quant Studio software. This software allows for large quantities of PCR reactions to occur sequentially so long as each plate has the proper input files linked to its barcode. The LIMS generates these files by scanning the Batch Experiment function as shown here.

There are two primary files required to run the PCR reaction. The first is a plate layout that dicates which chemicals are required for the reaction, which samples are in each well fo the plate, and what are the target and reference assays for the reaction. Depending on the assays, the chemicals and process may be different, so this must be defined in the setup files. The second file produced is simply the PCR plates barcode so it can be attached to the experiment.

Dosage Calling

The output of the qPCR process is a list of intensity values the the Quadrophenia dosage calling algorithm uses as input

The PCR reaction results in an output file that of the intensity measurements, or RQ values, for each of the samples. These RQ values following an exponential trend with distinct seperation based on the number of copies of the target assay. This is referred to as the samples' dosage; which is the goal of this entire process. The LIMS system automatically exports these files to the server once each plate in a job is finished.

The dosage calls are made using the Quadrophenia machine learning algorithm I developed. This algorithm uses the RQ values from the PCR exports to determine the dosage. It's vital that the data pipeline worked as planned throughout this process because about 10% of each plate consists of controls; or labeled samples. If the data is incorrectly stored, then these controls will be off and the dosage calling will fail. Once the dosage calls are made plots are produced to illustrate these calls. These plots are shown below and consist of two types. The type of the left mimics the plots that were previously generated in an attempt to call dosage by hand. The right plot is a simple scatter plot of RQ vs. predicted dosage.
